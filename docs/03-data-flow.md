## TL;DR (для комісії — 60 секунд читання)

**MyHealth-Europe нічого не збирає і нікуди не передає. Усе працює виключно на пристрої користувача.**

Користувач — це **єдиний** controller і **єдиний** processor своїх даних (у термінах GDPR). Проектна команда — і взагалі будь-хто, хто не є власником цього конкретного інстансу — не має доступу ні до одного байту користувацьких даних. Це не обіцянка у privacy policy, це архітектурний факт: на проектному боці немає бекенду, який міг би прийняти ці дані навіть гіпотетично.

Дані рухаються по системі так:

```
   КРОК 1 — Користувач     КРОК 2 — Імпорт у себе    КРОК 3 — Запит від AI
   ─────────────────────   ─────────────────────     ─────────────────────
   Іде на національний     Кладе скачаний файл       AI-агент просить scope
   e-health портал         у свій локальний          → користувач затверджує
   (eHealth UA, Digilugu,  MyHealth-Europe інстанс   → агент отримує токен
   Apple Health) — там     → парсинг, валідація,     обмеженого доступу
   натискає «експорт»      шифрування на локальному   → читає тільки те,
   → отримує файл          диску                     що дозволено
                                                     → запис у аудит-лог
```

**Ключове:** дані входять у систему **тільки** у вигляді файлу, який сам користувач свідомо приніс. Виходять — **тільки** у відповіді AI-агента, яку користувач читає сам. Жоден компонент проекту не «телефонує додому». Telemetry, аналітика, краш-репорти, оновлення — або повністю вимкнено, або opt-in із pseudonymous IDs.

## 2. Джерела даних (де живуть дані ДО того, як користувач їх принесе)

Ось вичерпний список джерел, які підтримує фаза 1 і які заплановані на фазу 2+. У всіх випадках користувач — **єдиний** актор, який вирішує, чи і коли експортувати.

### 2.1. Фаза 1 (поточний грант, 3 адаптери)

#### Джерело A — eHealth Україна

| Параметр | Значення |
|----------|----------|
| Володілець даних | Національна служба здоров'я України (НСЗУ) |
| Формат експорту | HL7 FHIR R4 bundle (JSON) |
| Спосіб отримання | Запит пацієнта через офіційний кабінет пацієнта (helsi.me або аналог) або письмовий запит до НСЗУ |
| Авторизація | Електронний цифровий підпис (ЕЦП) або Дія-підпис |
| Регуляторна основа | Закон «Основи законодавства про охорону здоров'я», ст. 39 (право пацієнта на медичну інформацію) |
| Технічні нюанси | Bulk-export може бути неповним для записів, створених до 2020; деякі провайдери досі не інтегровані з ЦБД eHealth |

**Що адаптер робить:** приймає файл, валідує проти FHIR R4 schema, нормалізує structure (NSZU-specific extensions → стандартні FHIR елементи де можливо), імпортує в локальний store.

#### Джерело B — Estonia Digilugu

| Параметр | Значення |
|----------|----------|
| Володілець даних | Estonian Health and Welfare Information Systems Centre (TEHIK) |
| Формат експорту | HL7 FHIR R4 (через ENA — Estonian National Adapter), CDA для legacy records |
| Спосіб отримання | digilugu.ee → авторизація через ID-card/Mobile-ID/Smart-ID → My data → Export |
| Авторизація | Estonian eID |
| Регуляторна основа | Estonian Personal Data Protection Act + EHDS Art. 3 (right to access) |
| Технічні нюанси | Найбільш зріла з e-health систем ЄС; чисті FHIR-bundles; добре документований; служить як reference-приклад |

**Що адаптер робить:** приймає файл, валідує, обробляє CDA→FHIR конверсію для legacy records, імпортує.

#### Джерело C — Apple Health

| Параметр | Значення |
|----------|----------|
| Володілець даних | Сам користувач (Apple — processor, не controller) |
| Формат експорту | XML (Apple's native export) + опційно FHIR (через iOS 16+ Health Records→Export to FHIR) |
| Спосіб отримання | iPhone → Health app → Profile → Export All Health Data → ZIP-файл |
| Авторизація | Face ID/Touch ID/passcode на пристрої користувача |
| Регуляторна основа | Apple Privacy Policy + GDPR (Apple зобов'язана надавати portability) |
| Технічні нюанси | XML-формат не FHIR-сумісний нативно — потрібна конверсія; FHIR експорт доступний лише для записів, отриманих із FHIR-сумісних провайдерів (US Health Records integration) |

**Що адаптер робить:** приймає ZIP, розпаковує, конвертує Apple Health XML у FHIR Observation/Condition resources, валідує, імпортує.

### 2.2. Фаза 2 (стрейтч / після основного гранту)

| Джерело | Формат | Спосіб | Складність |
|---------|--------|--------|------------|
| Google Health Connect (Android) | Custom data classes → FHIR convert | Android export інтент | Середня |
| Germany ePA (elektronische Patientenakte) | IHE XDS / FHIR (з 2025) | gematik портал з GesundheitsID | Висока (регулятор-важка) |
| France Mon Espace Santé | FHIR-based | DMP-портал | Середня |
| Poland IKP (Internetowe Konto Pacjenta) | HL7 CDA → FHIR convert | pacjent.gov.pl | Середня |
| Generic SMART-on-FHIR (live API) | FHIR через OAuth | Per-provider login | Висока (per-provider) |
| Garmin / Fitbit / Withings | Vendor JSON → FHIR convert | Vendor data export | Низька-середня |

### 2.3. Що проект НЕ робить з джерелами

- **Не реалізує scraping порталів.** Якщо джерело не має офіційного експорту — ми не пишемо клієнт, що логиниться і виносить дані. Це юридично і етично небезпечно.
- **Не зберігає credentials джерел.** Адаптер ніколи не отримує логін/пароль користувача до зовнішньої системи. Користувач сам логиниться, сам качає, сам приносить файл.
- **Не агрегує дані з різних користувачів.** Each instance бачить тільки свої дані. Немає cross-user аналітики, бо немає cross-user взагалі.

## 4. Trust boundaries (границі довіри)

```
ZONE 1: На пристрої користувача (HIGHEST TRUST)
   ├── MCP server process
   ├── UI client (browser → localhost)
   ├── Local store (encrypted SQLite)
   ├── Audit log (append-only)
   └── Consent gateway

ZONE 2: На тому ж пристрої, але інший процес (MEDIUM TRUST)
   └── AI agent (Claude Desktop, локальний Llama)
        Trust assumption: користувач сам обрав, сам встановив.
        Receives тільки те, що пройшло crom через consent gateway.

ZONE 3: Дистанційні AI-сервіси (USER-CHOSEN TRUST)
   └── Claude API, OpenAI API, EU-hosted Mistral
        Trust assumption: ЯКЩО користувач свідомо це обрав.
        Ми не можемо контролювати, що відбувається там.
        Все, що ми можемо — попередити в UI ("ця модель —
        cloud-based, дані підуть на сервер X").

ZONE 4: Зовнішні health-джерела (PRE-IMPORT, NO TRUST FROM US)
   └── eHealth UA, Digilugu, Apple Health
        Ми взаємодіємо з ними ТІЛЬКИ через файли,
        принесені користувачем. Жодних automatic
        викликів від нашої системи.

ZONE 5: Проектна команда і будь-хто за межами користувача (ZERO TRUST = ZERO ACCESS)
   └── Грибан Р., NLnet, Anthropic, downstream-впроваджувачі
        НЕ мають жодного доступу до даних будь-якого інстансу.
        Архітектурно неможливо, не lifestyle choice.
```

## 5. Аудит-лог (структурований запис кожного дотику до даних)

### 5.1. Що логується

Кожна з наступних дій створює immutable запис у локальному аудит-лозі:

| Подія | Поля |
|-------|------|
| `IMPORT` | timestamp, source (UA/EE/Apple), count_by_resource_type, file_hash, importer_version |
| `CONSENT_GRANTED` | timestamp, agent_id, scope, ttl, audit_id |
| `CONSENT_DENIED` | timestamp, agent_id, scope_requested |
| `CONSENT_REVOKED` | timestamp, agent_id, audit_id, revoked_by |
| `READ` | timestamp, agent_id, audit_id, scope, count_returned, resource_types |
| `EXPORT` | timestamp, target_format, count, user_initiated=true |
| `DELETE` | timestamp, resource_type, count, reason |

### 5.2. Властивості аудит-логу

- **Append-only** — записи не редагуються після створення.
- **Локальний** — лежить на тому ж пристрої, що й store.
- **Користувач — власник** — UI дозволяє переглядати, експортувати, але не редагувати.
- **AI Act compliance** — формат структурований так, щоб задовольнити вимоги transparency для AI-доступу до high-risk personal data.
- **GDPR Art. 30** — користувач може на запит exportувати весь лог як evidence of processing activities (якщо потрібно для DSAR).

## 6. Що НЕ існує в нашій системі (явні архітектурні no-go)

Цей розділ існує саме для комісії — щоб не лишилося сумнівів.

- **Немає бекенду проекту.** На стороні Грибана/команди немає сервера, який мав би endpoint типу `myhealth-europe.eu/api/upload`. Домен `myhealth-europe.eu` буде лише static-site (документація + завантаження бінарів/коду).
- **Немає telemetry за замовчуванням.** Якщо у фазі 2 з'явиться opt-in error reporting (Sentry або self-hosted GlitchTip), він буде:
  (а) вимкнений за замовчуванням,
  (б) обмежений stack-trace без user data,
  (в) pseudonymized,
  (г) повністю documented.
- **Немає cross-instance аналітики.** Ми не знаємо, скільки людей користуються софтом, окрім кількості GitHub clones і завантажень releases.
- **Немає update-каналу, що міг би push-нути код.** Оновлення — це звичайний git pull / docker pull, ініційований користувачем.
- **Немає key escrow.** Ключі шифрування генеруються локально, ніколи не покидають пристрій. Якщо користувач втрачає passphrase — він втрачає дані (з нашого боку відновити неможливо, і це навмисно).
- **Немає cloud-default.** Self-hosting cloud (наприклад, через docker compose на VPS користувача) — підтримуємо, але це окремий усвідомлений вибір, не default.

## 7. Що це означає для комісії NLnet/NGI

NGI Commons Fund оцінює за критеріями **excellence / impact-relevance-strategic / value-for-money**. Дата-флоу, описаний вище, прямо б'є по двох критеріях strategic-relevance:

1. **Internet as a commons.** Жоден компонент архітектури не вимагає централізованого посередника. Кожен користувач — повноцінний учасник, а не клієнт чужого сервісу. Це і є commons на технічному рівні.
2. **User agency.** Користувач — controller, processor, key holder, audit subject. Архітектура операціоналізує EHDS-права на рівні індивіда.

**Один-абзацна відповідь на питання «звідки беруться дані», для пітчу комісії:**

> Дані ніколи не потрапляють до нас. Кожен громадянин ЄС вже має право скачати свої медичні записи зі своєї національної e-health системи у форматі HL7 FHIR — це йому гарантує Регламент EU Health Data Space. MyHealth-Europe — це програма, яку громадянин запускає у себе на пристрої. Він приносить свій файл, програма імпортує його у свій локальний зашифрований сховище, і через стандарт Model Context Protocol надає AI-агенту контрольований доступ. Команда проекту не має ні бекенду, ні API, ні взагалі будь-якого технічного каналу, через який вона могла б побачити користувацькі дані. Це архітектурний факт, не політика.
